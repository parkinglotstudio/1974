# 누리(Nuri) 게임 — 적용 효과 목록

> 기준 버전: NuriGame.js v2.8  
> 마지막 업데이트: 2026-06-17

---

## 1. 모션 트레일 (달리기 중)

**함수:** `_renderTrail(ctx, entity)`  
**발동:** `runState === 'running'` 동안 매 프레임

- 진행 반대 방향으로 5개의 ghost 잔상을 순서대로 렌더
- 뒤로 갈수록 alpha 감소 (앞 0.30 → 뒤 0.06)
- `globalCompositeOperation = 'lighter'` (가산 합성) → 어두운 배경에서 빛나는 잔광 느낌
- 잔상 간격: 7px, 현재 프레임의 `_entBuf`를 그대로 재사용해서 퍼포먼스 손실 없음

---

## 2. 발밑 모래 먼지 (달리기 중)

**함수:** `_updateDust(dt)` / `_renderDust(ctx)`  
**발동:** `runState === 'running'` 동안 매 프레임, 초당 약 18개 스폰

- 캐릭터 `_entBuf`의 하단 22% 픽셀에서 색상 샘플링 후 입자 생성
- 입자가 진행 반대 방향(-35px/s)으로 흩날리며 중력(120px/s²) 낙하
- 수명 0.4~0.9초, 수명 60% 이후부터 fade out
- 2×2 그리드 정렬로 픽셀아트 느낌 유지

---

## 3. 모래시계 스트림 (클릭 이동)

**함수:** `_renderPixelMove(ctx, b)`  
**발동:** 클릭 후 캐릭터 이동 시

캐릭터 픽셀이 4단계로 흘러서 목적지에 재조립되는 모래시계 이동 연출.

| 구간 | trav 범위 | 동작 |
|---|---|---|
| 출발 | 0 → 0.25 | 각 픽셀이 수집점(gatherX/Y)으로 모임 |
| 스트림 | 0.25 → 0.75 | 직선으로 방출점(releaseX/Y)까지 흐름 |
| 도착 | 0.75 → 1.0 | 목적지 형태로 재조립 (cubic ease) |

- 파티클별로 `ff`(출발 딜레이)가 랜덤 → 픽셀이 한꺼번에 움직이지 않고 물결처럼 흐름
- **거리 비례 duration:** 최소 350ms ~ 최대 1000ms (화면 폭 960px 기준 1400ms 비율)
  - 가까운 클릭 = 빠르게, 먼 클릭 = 느리게

---

## 4. 스파이크 색상 (클릭 이동 스트림 중)

**위치:** `_renderPixelMove` 스트림 구간 내부  
*(현재 비활성 — ECG 파형 제거로 같이 비활성됨)*

> ECG 파형이 튀는 순간 각 픽셀에 +140R, +180G, +255B 를 더해 파란색으로 번쩍이는 효과.  
> ECG 파형 복구 시 자동으로 함께 활성화됨.

---

## 5. 출발 / 도착 플래시 (클릭 이동)

**위치:** `_renderPixelMove` 내 플래시 블록  
**상수:** `FLASH_MS = 200`

- 이동 **시작** 200ms: 출발 위치 픽셀이 흰색으로 번쩍이며 사라짐
- 이동 **완료** 200ms: 도착 위치 픽셀이 흰색으로 번쩍이며 나타남

---

## 6. 오비터 (항상)

**함수:** `_renderOrbits(ctx, frontPass, activeEnt)`  
**발동:** 항상 (아이들·달리기·클릭이동 무관)

- 캐릭터 허리 높이(ph × 0.38) 기준으로 5개 오비터가 타원 궤도로 공전
- 각 오비터는 네모↔원형 사이를 `morphT`로 smoothstep 모프
- sin(angle) > 0 = 앞 패스, < 0 = 뒤 패스로 나눠서 캐릭터 앞뒤 입체감 표현
- `globalCompositeOperation = 'lighter'` 가산 합성, 핑크·시안 2색 교차

## 상수 요약

| 상수 | 값 | 설명 |
|---|---|---|
| `CHAR_SCALE` | 1.0 | 캐릭터 렌더 배율 (원본 해상도) |
| `RUN_SPEED` | 160 px/s | 조이스틱 달리기 속도 |
| `FLASH_MS` | 200 ms | 클릭 이동 플래시 지속 시간 |
| `WAIST_SCALE` | 0.05 | 스트림 허리 폭 계수 |
| `PM_DURATION` | — | 거리 비례로 동적 계산 (350~1000ms) |

# 누리 (NURI) — 작업 규칙

## 토큰 절약 원칙 (필수)

1. **전체 파일 Read 금지** — `NuriGame.js`처럼 큰 파일은 절대 통째로 읽지 않는다.
   먼저 Grep으로 대상 메서드/줄 번호를 찾고, 그 구간만 `offset`/`limit`으로 Read한다.
2. **스크린샷은 1번만 시도** — `preview_screenshot`이 타임아웃되면 재시도하지 말고
   즉시 `preview_eval`(state/pixel 직접 확인)로 전환한다. 백그라운드 탭에서는
   `requestAnimationFrame`이 멈춰 스크린샷이 항상 실패하는 환경 제약이 있다.
3. **검증은 핵심 1~2개만** — 수정 하나당 전체 흐름을 풀세트로 재검증하지 않는다.
   바뀐 부분에 영향받는 가장 중요한 동작 1~2개만 `preview_eval`로 확인하고 끝낸다.
4. **삭제용 Edit은 grep으로 범위 확정 후 한 번에** — 코드를 옮길 때 old_string에
   삭제할 코드 전체를 다시 적어야 하므로, Read는 삭제 대상 구간만 정확히 좁혀서 한다.

## 아키텍처 — 모듈 분리 현황

`NuriGame.js`는 오케스트레이터만 남기고 책임별로 `scripts/core/*.js`, `scripts/render/*.js`로 분리 중.

| 모듈 | 파일 | 상태 |
|---|---|---|
| 격자파동 전투 엔진 | `core/GridWaveSystem.js` | 완료 |
| 공유 상수 | `core/constants.js` | 완료 (`HOLD_THRESHOLD`) |
| 메시지(트리거 안내) | `core/MessageSystem.js` | 완료 |
| 탄창 슬롯(궤도/잠금/소모) + 조준선 | `core/AmmoSystem.js` | 완료 |
| 임팩트 파티클/쇼크링/랜드플래시 | `render/EffectsRenderer.js` | 예정 |
| 캐릭터 렌더(림라이트/버퍼/트레일) | `render/CharacterRenderer.js` | 예정 |
| 입력(조이스틱/포인터) | `core/InputController.js` | 예정 |
| 이동 상태머신(공중/달리기) | `core/MovementController.js` | 예정 |
| 픽셀무브 블렌드 | `render/PixelMoveBlend.js` | 예정 |
| 배경/카메라 | `render/BackgroundLoader.js` | 예정 |

**분리 패턴**: 새 클래스는 생성자에서 `game`(NuriGame 인스턴스) 참조 하나만 받고,
필요한 상태는 `game._player`, `game._cameraX` 등으로 접근. NuriGame에는
`this.xxx = new XxxSystem(this)` 인스턴스 + 얇은 호출만 남긴다.

## 데이터 흐름
CSV(`data/*.csv`) → `main.js`에서 `loadCsvTable`/`loadParamTable` → `engine.data.*` →
각 시스템 생성자에서 읽어 내부 상태로 변환. 새 기능 추가는 가능한 CSV 행 추가로 해결.

## 알려진 환경 제약
- 프리뷰 탭이 백그라운드면 rAF가 멈춰 `preview_screenshot`이 항상 타임아웃된다.
  스크린샷이 필요하면 사용자에게 직접 확인을 요청하거나 `preview_eval`로 우회한다.
- `NuriGame.js` import는 캐시버스트 쿼리(`?v=X.X`)를 쓴다 — 모듈 수정 시 버전 올리는 것을 잊지 말 것.
