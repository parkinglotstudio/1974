## 채널 ↔ 게임 API 계약 (postMessage)

**모든 Sand Engine 게임은 이 계약을 반드시 구현해야 한다.**
채널이 게임을 iframe으로 실행할 때 유일한 통신 수단.

### embed 모드 감지
```js
const isEmbed = window !== window.top;
```

### 게임 → 채널
```js
// 게임 준비 완료
window.parent.postMessage({ type: 'ready' }, '*');

// 점수 전달
window.parent.postMessage({ type: 'score', value: 100 }, '*');

// 게임 완료
window.parent.postMessage({ type: 'complete', result: { score, time, grade } }, '*');

// 게임 종료 요청 (뒤로가기 등)
window.parent.postMessage({ type: 'exit' }, '*');
```

### 채널 → 게임
```js
// 게임 시작
iframe.contentWindow.postMessage({ type: 'start' }, '*');

// 일시정지
iframe.contentWindow.postMessage({ type: 'pause' }, '*');

// 재개
iframe.contentWindow.postMessage({ type: 'resume' }, '*');
```

### GameRunner (핵심)
채널이 게임을 실행하는 유일한 방법.

```js
// GameRunner 역할
// 1. games/{id}/game.json 로드 → 게임 메타 확인
// 2. iframe 생성 → games/{id}/index.html 로드
// 3. postMessage 수신 → 채널 상태 업데이트
// 4. 게임 complete 수신 → 챕터 진행 반영, 아바타 진화 트리거

class GameRunner {
    run(gameId)      // 게임 실행
    pause()          // 일시정지
    resume()         // 재개
    exit()           // 게임 종료 → 챕터로 복귀
}
```

통신 API는 `CLAUDE.md`의 postMessage 계약 참조.

## 채널 API 구현 규칙

**모든 게임의 `index.html`은 아래를 반드시 구현한다.**

```js
// 1. embed 모드 감지
const isEmbed = window !== window.top;

// 2. 채널 명령 수신
window.addEventListener('message', (e) => {
    if (!e.data?.type) return;
    if (e.data.type === 'start')   game.start();
    if (e.data.type === 'pause')   game.pause();
    if (e.data.type === 'resume')  game.resume();
});

// 3. 준비 완료 알림
window.addEventListener('load', () => {
    if (isEmbed) window.parent.postMessage({ type: 'ready' }, '*');
});

// 4. 결과 전송
function onGameComplete(result) {
    if (isEmbed) {
        window.parent.postMessage({ type: 'complete', result }, '*');
    }
}
```

전체 API 계약은 `CLAUDE.md` 참조.

# 엔진 기능과 규칙 — Sand Engine이 할 수 있는 것·세팅 방법 (새 게임/씬 만들 때 필독)
> 2026-06-04 기준. 이 문서는 **엔진의 고유 기능과, 새 씬/게임을 세팅할 때 반드시 지킬 규칙**을 정의한다.
> 원칙: **엔진 = 로직(코드) / 게임 = 데이터(CSV·씬·픽셀 에셋)**. 게임은 값만 세팅하고, 매번 로직을 새로 짜지 않는다.

---

## 1. 레이어 모델 (근경/원경 분리 = 공간감의 핵심)

| 레이어 | 용도 | parallax | 내용 |
|---|---|---|---|
| **L0** | 원경(far) | **0.30** | 풀컬러 스카이라인 (scanline). 카메라보다 느리게 스크롤 |
| **L1** | 근경(near) = **월드 평면** | **1.00** | 검은 실루엣(+알파). 캐릭터와 같은 깊이, 월드폭 = 이 레이어 폭 |
| **L2** | 캐릭터/오브젝트 | 1.00 | EntitySystem |
| **L3** | 전경 | 1.40 | (현재 미사용) |

- **렌더 순서**: L0 → L1 → L2 → L3 → FX(glow→lighting→rim→vignette).
- **패럴랙스 공식**: 화면에 `layer[srcX .. srcX+W]` 표시, `srcX = clamp(cameraX × parallax, 0, layerW − W)`. 확대 없음(1:1 슬라이스).
- **원경 폭 규칙**: `원경폭 ≥ 화면폭 + (월드폭 − 화면폭) × parallax` 여야 끝에서 안 멈춤. (현재 1706 월드 + p0.30 → 890px 이상 필요, 원경은 1706이라 충분)

## 2. 근경/원경 제작 파이프라인 — `tools/split_bg_layers.py`
샘플 명명 (era 예 `1960_1`): `{era}_1_1_1.png`=근경(배경제거 실루엣), `{era}_1_1_1_1.png`=원경(풀컬러).
- **원경**: 리사이즈(1706×960) → **메디안컷 적응형 팔레트**로 scanline. (빈도순 팔레트는 노을·불빛 같은 흩어진 색을 버려 파랗게 뭉개짐 — 금지)
- **근경**: 색을 **전부 검정(#000000)** 으로, **불투명도(파일 알파)는 유지** → 검은 실루엣. 엔진 팔레트가 `#rrggbbaa` 8자리 알파를 지원하므로 가장자리가 부드럽게 페이드.
- 출력: `pixels/objects/{era}_near.json`, `{era}_far.json`.

## 3. 시그니처 FX (전부 데이터로 조절 — `fx_params.csv`)

### 낮↔밤 60초 사이클 — **콘트라스트가 핵심**
- "밤 = 균일하게 어둡게"가 아니라 **어두운 데 더 어둡게 + 밝은 데 더 밝게**.
- **ambient(곱하기 어둠) × k** → 그림자·어두운 건물 가라앉음 (`LightingSystem`, multiply 합성).
- **glow 블룸 × k** → 창문·불빛 도드라짐 (가산).
- 이 둘의 합 = 콘트라스트 S커브. **중앙 방사형 key 라이트는 쓰지 않는다** (밝은 원경에 흰 wash 만듦).
- 어둠색 = 원경 대표톤(`_sampleBgTone`, **L0 샘플**)의 깊은 버전 → 그림자가 배경 톤을 띰.

### 글로우 블룸 (`GlowSystem`)
- **밝은 불빛만 번지게**: `bloom_threshold ≈ 205` (낮으면 밝은 원경 전체가 떠서 뿌예짐).
- **저해상도(1/3) 빌드 필수**: 큰 배경 레이어(2160~4320폭) 풀해상도 빌드는 ~700ms 프리징. `_buildBloom`이 1/DS로 빌드 후 업스케일.
- 캐릭터(L2) 발광은 끈다: `removeEmissiveIndex(10)` — 캐릭터 팔레트 idx10이 엔진 예약 발광색과 충돌해 노란 halo 생김.

### 캐릭터 역광 림 (게임측 `_renderCharRim`)
- env(림 색) = **원경(L0) 픽셀별 색** + floor(bgTone). 밤(캐릭터 밝기<`rim.fade_threshold`)부터 페이드인.
- 엔진 `RimLightSystem`은 OFF(방향 뭉개짐 방지), 게임측 단일 림 사용.

### 근경 역광 림 (`_renderNearRim`) — 평평한 실루엣에 입체감
- 원경(L0) 빛이 근경(L1) 실루엣 가장자리를 가산으로 감쌈 → 역광 받는 전경.
- **맵별 배율**(`maps.csv near_rim`): 밝은 원경 맵은 낮게(네온 윤곽 방지), 어두운 맵은 높게.
- 비용: 반해상도 처리 + ImageData 1회 합성.

### 캐릭터 색 리빌 (`_renderCharReveal`)
- 밤에 캐릭터를 고유색의 **20~35%**(`reveal.night_base`+`night_lum_scale`×배경밝기)로 가라앉힘. 투명도가 아니라 밝기만 낮춤(비침 방지).

## 4. 성능 규칙 (반드시 지킬 것)
- **`getImageData`는 GPU→CPU 동기 스톨**(크기 무관 비쌈). 프레임당 호출 최소화 / 공유(`_readCharRegion`처럼 1회 읽어 여러 FX 공유).
- **전환(transition) 중엔 캐릭터 FX·근경림·모래알 스킵** — 모래가 화면 덮어 안 보이는데 getImageData만 쌓임 → 6fps 추락 원인이었음.
- **블룸은 저해상도 빌드 + 맵 전환 시 원경(L0)만 재빌드**. 검은 근경(L1)은 블룸 불필요.
- **정적(scanline) 배경은 1회만 래스터**(`_static` 플래그, EntitySystem이 캐시). 카메라 이동은 슬라이스만(재래스터 X).
- 픽셀당 `ctx.fillRect` 금지 → ImageData 버퍼에 써서 `drawImage` 1회.

## 5. 데이터 주도 규칙 (게임 = CSV)
- 코드에 매직넘버 박지 말 것. `engine/data/Csv.js`로 로드, 게임은 상수 대신 테이블 값 사용(없으면 코드 fallback).
- 테이블: `maps.csv`(맵·순환·맵별 림), `characters.csv`(스탯), `animations.csv`(fps), `fx_params.csv`(시그니처 수치).
- **씬(scene.json) = 배치**(무엇을 어디에), **CSV = 데이터/스탯/튜닝**, **pixels/*.json = 그림 에셋**. 역할 섞지 말 것.

## 6. 배포 규칙 (엔진 소스 비노출)
- `npm run build`(build.mjs) → `dist/`: **`app.min.js` 한 파일로 번들·압축, 소스맵 없음, engine/·scripts/·samples/ 원본 미포함**.
- `vercel.json`: buildCommand `npm run build`, outputDirectory `dist`. main 푸시 → 자동 빌드·배포.
- 검증: `dist`에 engine/ 폴더·`.map` 없음, `app.min.js`에 엔진 경로/주석 0건.

  `maps.csv`의 `cycle_order >= 0`(순환에 실제 쓰이는 맵)인 asset 접두어 파일만 `dist`에 포함되도록
  `build.mjs`가 자동 필터링함(`pixels/characters/`는 전부 포함). 새 맵을 순환에 추가/제외할 때는
  `maps.csv`의 `cycle_order`만 맞게 관리하면 됨 — `build.mjs`를 따로 손댈 필요 없음.
  (2026-06-11: 이전 배포 실패 원인 = dist 150MB(미사용 맵 픽셀 ~67.5MB + 사용자 원본 소스
  43MB가 `pixels/objects/`에 잘못 위치) 초과. 미사용 맵 제외 + 원본 파일을 `samples/`로 이동 →

# 안티그래비티 필독 — 모래엔진 기본 상식과 흔한 실수

> 작성: Claude · 2026-06-08  
> **이 문서는 새 게임이나 기능을 만들기 전에 반드시 읽을 것.**  
> 엔진을 잘못 이해하면 코드가 동작해도 성능이 망가지거나 엔진 구조를 우회하게 된다.

---

## 1. 핵심 대원칙 (가장 중요)

> **엔진 = 로직(코드) / 게임 = 데이터(CSV·씬JSON·픽셀에셋)**

게임 코드는 엔진이 제공하는 인터페이스만 사용하고, 로직을 새로 짜지 않는다.  
엔진이 이미 하는 일을 게임 코드에서 다시 구현하면 → 버그, 성능 저하, 유지보수 불가.

---

## 2. 픽셀 데이터는 반드시 "변환 도구"로 만든다

### ❌ 절대 하지 말 것
```js
// 코드로 직접 픽셀 배열 생성 — 절대 금지
const pixels = [];
for (let y = 0; y < 252; y++)
    for (let x = 0; x < 252; x++)
        pixels.push([x, y, palIdx]);
entity._pixels = pixels;
```
이렇게 하면:
- JS에서 63,504개 배열 생성 = 느림
- 엔진 정적 최적화(한 번만 래스터) 완전 무효화
- 매 프레임 재처리

### ✅ 올바른 방법
픽셀 데이터는 **오프라인 변환 도구**로 JSON 파일로 만들어 두고, 씬에서 에셋으로 로드한다:
```
tools/gif_to_sprite.py       → 캐릭터 스프라이트 JSON
tools/png_to_pixels.py       → 배경/오브젝트 픽셀 JSON
tools/split_bg_layers.py     → 근경/원경 레이어 JSON
```
이렇게 만든 JSON을 씬에서 `asset` 필드로 참조하면 엔진이 알아서 로드·최적화.

---

## 3. 씬 로더를 게임 코드에서 재구현하지 말 것

### ❌ 잘못된 방법
```js
// PlayScene.js에서 씬을 수동으로 파싱하는 함수를 새로 만듦 — 금지
async applyCustomScene(engine, sceneJSON) {
    for (const def of sceneJSON.entities) {
        engine.entities.add(def.id, cfg);
    }
}
```
골목길의 `main.js`에 이미 `applyAssetScene()`이 완성돼 있다.  
씬 JSON 포맷을 지키면 엔진이 알아서 로드한다. 중복 구현 금지.

### ✅ 올바른 방법
씬 JSON(`scenes/main.scene.json`)만 올바르게 작성하면 된다.  
에셋 참조는 `asset` + `assetCategory` 필드로:
```json
{
  "id": "board_bg",
  "asset": "board_bg",
  "assetCategory": "objects",
  "layer": 1,
  "x": 9, "y": 84, "pw": 252, "ph": 252
}
```

---

## 4. 엔진 FX 초기화 블록을 절대 건드리지 말 것

`GolmokGame.js`의 `onEnter()` 상단에 있는 FX 초기화 블록:
```js
engine.lighting.setAmbient(0.0);
engine.glow.removeEmissiveIndex(10);   // ← 이거 지우면 밤에 캐릭터 노란 halo
engine.rim.disable();                  // ← enable()로 바꾸면 림이 2중 렌더됨
engine.fog.enableLayerFog({...});
engine.vignette.setPreset('warm');
```
이 블록은 **절대 삭제/수정 금지**. 이유가 주석에 설명돼 있다.  
2026-06-07: 이 블록을 삭제했다가 밤 halo·림 이중렌더·비네트 소실 발생. 직접 복구함.

---

## 5. 엔진 레이어 구조

| 레이어 | 이름 | 용도 | parallax |
|--------|------|------|----------|
| L0 | 원경(far) | 하늘·스카이라인 등 뒷 배경 | 0.30 |
| L1 | 근경(near) | 전경 실루엣·건물 등 앞 배경 | 1.00 |
| L2 | 게임 | 캐릭터·인터랙티브 오브젝트 | 1.00 (스크린 좌표) |
| L3 | UI | 점수·HUD 등 화면 고정 요소 | 1.40 |

- L2는 **스크린 좌표**로 렌더됨 (cameraX가 빠진 위치).
- L0·L1은 **월드 좌표** → cameraX에 parallax 곱해서 슬라이싱.

---

## 6. `_static` 최적화 — 정적 배경은 1회만 래스터

배경처럼 움직이지 않는 엔티티는 `_scanline` 포맷을 쓰면 엔진이 자동으로 정적 레이어로 처리 → 첫 프레임에만 그리고 이후 스킵 (= 성능 핵심).

동적 오브젝트(블록, 캐릭터 등)는 `pixels` 또는 `frames` 포맷 → 매 프레임 갱신.

---

## 7. doubledown_pixel이 잘못한 것 (구체적 사례)

`games/doubledown_pixel/` 폴더의 현재 코드에서 발견된 문제:

1. **`BoardRenderer.js`**: 블록마다 `entity._pixels`를 JS 코드로 동적 생성 → 변환 도구 사용 안 함, 성능 최악.
2. **`PlayScene.js`의 `applyCustomScene()`**: 씬 로더 재구현 → `main.js`의 `applyAssetScene()` 무시.
3. **`index.html`**: `console.log`마다 `/api/log`로 네트워크 요청 → 디버그 코드가 프로덕션에 남아있음. 반드시 제거.
4. **`hudExternalStore.js`**: React/Zustand 스타일 외부 상태 관리 → 엔진 데이터 흐름(CSV→씬→엔티티) 완전 무시.
5. **SceneJSON의 `board_bg` 엔티티**: asset 없이 씬에만 선언 → 실제 픽셀 데이터가 씬 JSON에 없어 빈 엔티티.

---

## 8. 새 게임 만들 때 올바른 순서

1. `games/새게임/game.json` 작성 (해상도·레이어·씬 목록)

# 데이터 테이블 ↔ 엔진 값 연동 (CSV 시스템)

> 작성: Claude · 2026-06-10 · 기준 커밋: golmok `data/*.csv` + `engine/data/Csv.js`
> **이 문서 하나로 "시그니처 효과 수치가 어떻게 코드와 연결돼 있는지" 전부 파악 가능.**
>
> 대원칙: **엔진 = 로직(코드) / 게임 = 데이터(CSV)**.
> 코드에 매직넘버를 박지 않고 표(CSV)에서 읽는다. 나중에 에디터 툴이 이 CSV(또는 슬라이더)만
> 편집하면 게임 룩·스탯·연출이 바뀐다. 코드 수정 불필요.

---

## 1. 한눈에 — 전체 흐름

```
games/golmok/data/*.csv          ← 게임 데이터 (값)
        │  fetch (no-store)
        ▼
engine/data/Csv.js               ← 엔진 범용 로더 (로직)
   loadCsvTable() / loadParamTable()
        │
        ▼
main.js init()
   game._engine.data = { maps, characters, animations, fx }
        │
        ▼
GolmokGame._loadConfig(engine)   ← onEnter에서 1회
   ① 모듈 let 상수 덮어쓰기
   ② this._p 인라인 수치 채움
   ③ 애니 fps/loop override
   ④ 배경 순환목록 + 맵별 근경림 배율 구성
        │
        ▼
onUpdate / _render*()            ← 매 프레임 this._p.* · 모듈 상수를 읽어 FX 적용

[에디터 라이브] applyFxParam(key, value)  ← 슬라이더가 호출 → 다음 프레임 즉시 반영(저장 X)
```

CSV 로드 실패해도 게임은 안 죽는다 → 코드 내 **fallback 기본값**으로 동작.

---

## 2. 엔진 로더 — `engine/data/Csv.js`

범용 CSV 유틸. **CSV 규칙**: 첫 비주석 줄 = 헤더, `#` 줄 = 주석, 빈 줄 무시, 쉼표 구분(값에 쉼표 있으면 `"..."`로 감쌈).

두 가지 테이블 형태:

| 클래스 | 형태 | 용도 | 접근 |
|---|---|---|---|
| **`CsvTable`** | 행 = 레코드 | maps / characters / animations | `.all()` · `.byId(field,val)` · `.filter(field,val)` · `.indexBy(field)` |
| **`ParamTable`** | key,value | fx_params | `.num(key,def)` · `.bool(key,def)` · `.str(key,def)` · `.meta(key)`→{min,max,desc} |

- `loadCsvTable(url)` → `CsvTable`
- `loadParamTable(url)` → `ParamTable`
- `ParamTable`은 슬라이더용 `min/max/desc` 메타를 따로 보관(코드는 `value`만 읽음).

---

## 3. CSV 테이블 4개 — `games/golmok/data/`

| 파일 | 행 = | 핵심 열 | 소비처 |
|---|---|---|---|
| `maps.csv` | 배경 1장 | id, asset, world_w/h, **cycle_order**(순환순서, -1=제외), **near_rim**(맵별 근경 역광 배율) | `_loadConfig` ② ④ |
| `characters.csv` | 캐릭터 1명 | id, asset, pw/ph, spawn_x/y, **move_speed**, layer | `_loadConfig` ③ |
| `animations.csv` | 캐릭터 상태 | char_id, state, frames, **fps**, **loop** | `_loadConfig` ③ |
| `fx_params.csv` | 파라미터 1개 | key, **value**, min, max, desc | `_loadConfig` ① ⑤ + `applyFxParam` |

### 역할 분담 (섞지 말 것)
- **CSV** = 타입·스탯·튜닝 (이동속도, 애니 fps, FX 수치, 맵 순환).
- **씬(scene.json)** = 배치 (어떤 맵·캐릭터를 어느 좌표에). **좌표는 씬이 우선**, CSV는 기본/스탯.
- **픽셀 JSON(pixels/)** = 실제 그림 자산.

### 주의 — `animations.csv`의 `frames` 칸은 참고용
프레임 데이터의 진실은 **스프라이트 stateDef(JSON)**. CSV의 `frames`는 개수 메모일 뿐이고,
로드 시 실제로 override되는 건 **fps/loop만**.

---

## 4. ★ `fx_params.csv` = "효과 모아놓은 데이터"

시그니처 FX **모든 튜닝 수치를 한 표로** 뺐다. 1행 = 파라미터 1개(`key,value,min,max,desc`).
min/max는 에디터 슬라이더 범위용 메타(코드는 value만 읽음).

8개 그룹:

| 그룹 | 무엇 | 대표 키 |
|---|---|---|
| `daynight.*` | 낮↔밤 사이클 | `period_s`(주기 60s), `char_bright_min`(밤 최저 캐릭터 밝기 0.20) |
| `rim.*` | 캐릭터 역광 림 | `fade_threshold`, `peak`, `floor_mult`, `width_base`, `dir_base` |
| `nearrim.*` | 근경 실루엣 역광 림 | `base`(낮), `night`(밤) — 맵별 배율은 `maps.csv near_rim` |
| `reveal.*` | 캐릭터 색 리빌(밤 어두워짐) | `night_base`, `night_lum_scale`, `blend` |
| `light.*` | 전역 조명 | `ambient_max`, `key_intensity_max`, `vignette_max`, `glow_max`, `bloom_threshold`, `fog_max` |
| `bg.*` | 배경 순환/전환 | `interval_ms`, `trans_ms`, `mote_rate` |
| `intro.*` | 진입 인트로 | `ms` |
| `sand.*` | 좌우 이동 모래 스크롤 | `band_max`, `grain`, `settle_ms`, `buildup_ms`, `fall_rate`, `gravity` |

> 각 수치의 **현재 적용값·의미·튜닝 노하우**는 `docs/골목길_시그니처FX_수치.md` 참조.
> 이 문서는 **연동 구조**, 그 문서는 **값 자체**를 담당.

## 5. 게임측 소비 — `GolmokGame.js`

### `_loadConfig(engine)` (onEnter 1회, line ~169)
`engine.data`에서 값을 읽어:
1. **모듈 `let` 상수 덮어쓰기** — `BG_TRANS_MS`, `SAND_*`, `INTRO_MS` 등. 사용처 코드는 그대로 상수를 읽으므로 수정 불필요.
2. **배경 순환목록** = `maps.csv`에서 `cycle_order >= 0` 행을 정렬 → `BG_LIST`.
3. **캐릭터 스탯** = `characters.csv` `move_speed` → `MOVE_SPEED`.
4. **애니 타이밍** = `animations.csv`의 fps/loop → 상태머신 `asm.states[state]`에 override.
5. **인라인 수치** = `fx_params.csv` → `this._p {}` (dayPeriod, rim*, reveal*, ambMax, glowMax, bloomThreshold, nearRim* …). render/onUpdate가 `this._p.*`를 매 프레임 읽음.
6. **맵별 근경림 배율** = `maps.csv near_rim` → `this._mapRim{}`.

> `light.bloom_threshold`를 바꾸면 **블룸 캐시 재빌드 필요** → `engine.glow._bloomThreshold` 설정 후
> `layer._bloomReady = false`로 강제 재빌드. (이 패턴 유지할 것.)

### `applyFxParam(key, value)` (에디터 라이브 슬라이더, line ~604)
- `fx_params.csv`의 key → 소스값(`this._p.*` 또는 모듈 let)을 **즉시 갱신**. 다음 프레임 반영.
- **저장은 안 함** — 확정된 값은 사람이 `fx_params.csv`(또는 코드 fallback)에 수동 반영해야 영구화.
- **`_loadConfig`와 1:1 매핑**이어야 함 → fx 파라미터를 추가/이름변경하면 **세 곳을 같이 고칠 것**:
  1. `fx_params.csv` (행 추가)
  2. `_loadConfig` `this._p` 매핑 (+ 필요시 fallback 기본값)
  3. `applyFxParam` switch case

# 가로(시네마) 성능 점검 — 엔진 리뷰

> 작성: Claude (Opus) · 2026-06-06
> 증상: 가로 시네마 해상도(1760×740)로 키우자 **프레임 급락**.
> 결론: 해상도 비례 비용 중 **CPU측(거대 캔버스 + readback 스톨)이 주범** → 엔진/게임에서 3가지 수정. GPU 패스는 2.6배라도 문제 아님.

---

## 찾은 원인 & 수정 (커밋됨)

### 1. ★ 레이어 캔버스 폭 과대 (가장 큼)
- `LayerSystem`이 캔버스 폭 = `viewWidth × mult`([4,8,1,8]). 가로 1760뷰 → **L1/L3 = 14080폭** (배경은 2433폭뿐인데).
- 매 프레임 clear/composite가 이 거대 캔버스를 처리 → 급락.
- **수정**: `LayerSystem.MAX_CW = 5120` 상한 캡. L1/L3 14080→5120 (**2.8배↓**). 세로엔 영향 거의 없음.
- ⚠️ 배경 월드폭이 5120 넘으면 `MAX_CW`도 올릴 것(안 그러면 스크롤 잘림).

### 2. ★ 근경림 readback 스톨 (_buildFxSrc)
- 근경 역광림 FX가 매 프레임 `getImageData`로 **풀폭 × 반해상 ≈ 1.14M** 픽셀을 GPU→CPU 읽음(스톨). 고해상도일수록 GPU 대기열이 길어 스톨이 더 비쌈.
- **수정**: 근경림 소스를 **반해상→쿼터해상**(`hw=W>>2, hh=H>>2`). readback **1.14M→0.41M (2.8배↓, 960×540보다도 작음)**, 림 JS 루프 4배↓.
- **캐릭터 영역 readback은 그대로 풀해상** 유지 → 캐릭터 화질 영향 없음. 근경 역광림만 살짝 더 부드러운 글로우(밤에만 나오는 서브틀 FX라 영향 미미).

### 3. 원경 크로스페이드가 L0를 비정적화
- 크로스페이드(_img 2장)가 L0를 `_static=false`로 만들어 **매 프레임 재래스터**.
- **수정**: `FAR_XFADE=false`(OFF) → 원경 정적 복귀. 켜려면 플래그만 ON(그땐 비용 감수).

---

## 남은 해상도 비례 비용 (현재 수용 가능)
- **L2(캐릭터) flush `putImageData`** 풀캔버스(1760×740≈5MB/프레임). 단순 대역폭(수백 MB/s) — GPU 대역폭 충분, 스톨 아님. 정 필요하면 dirty-rect로 캐릭터 bbox만 쓰게 최적화 가능(엔진 변경, 테스트 필요).
- **풀캔버스 GPU 패스**(lighting multiply·glow·fog·vignette·composite): 면적 2.6배지만 전부 GPU 합성 → 저렴.
- **전환(sandburst)**: `_tr` 중 풀캔버스 `getImageData`+픽셀루프+`putImageData`를 매 프레임(2초). **전환 순간만** 무거움. 거슬리면 G스텝 키우거나 `BG_TRANS_MS` 줄이기.

## 권장
1. 지금 **1760×740 그대로 테스트** — 위 3수정으로 평상시 비용이 960×540 수준 근처로 내려갔을 것.
2. 그래도 부족하면 순서대로: ① 해상도 한 단계↓(예 1536×648, 캐릭터 40%) ② 전환 비용 최적화 ③ L2 dirty-rect flush.
3. 근경림 쿼터해상이 너무 흐리면 `hw=W>>2`→`W>>1`(반해상)로 되돌리되 readback 비용 복귀.

## 4.5 로얄매치 톤 스킨 패스 (2026-06-12 추가)

디렉팅: **색·버튼 느낌을 로얄매치 수준으로** (레이아웃은 유지). 버튼 아트·아이콘은 Oppa가
그림으로 제공 예정 — 그 전까지 절차 생성 베벨, 제공되면 `image` 위젯으로 교체.

### 엔진 추가 기능
- **PixelFont 아웃라인** — 마스크 다중 스탬프로 청키한 픽셀 외곽선. `{ outline:'#1B2A4A', outlineW:2 }`.
  내부가 마스크/틴트 캐시 2단으로 분리됨.
- **컬러 램프(skin)** — `ui.setRamps(palette.ramps)`. 램프 = [하이라이트, 본색, 베벨, 아웃라인] 4톤.
  위젯 `"skin":"gold"` → 글로시 베벨 버튼/패널 (아웃라인→본색→상단 하이라이트→하단 베벨,
  눌림 = 하이라이트 제거+침하). skin 없으면 기존 플랫 렌더 (하위 호환).
- **pop 트윈(주스)** — 버튼 액션 발화 시 1→1.08→1 스케일 160ms 자동.
- **image 위젯** — `{ "type":"image", "src":"ui/icon.png" }` 사용자 제공 그림 직접 렌더(픽셀레이트),
  로드 완료 시 자동 재래스터. **아이콘/버튼 아트 교체 진입점.**
- 게이지 skin 렌더 (트랙 어둡게 `track` 오버라이드 가능).

### 게임 데이터 변경
- `palette_square.json`에 `ramps` 8종 (gold/green/blue/red/purple/cream/navy/slate — 로얄매치 톤).
- `ui_lobby.json`: **이벤트/재화/유료/마일스톤 슬롯 전부 제거**(디렉팅), 딥블루 배경 + skin 적용.
- `main.js`: `engine.ui.setRamps(palData.ramps)` 배선.

### ⚠️ 폰트 크기 규칙 (중요)
**Galmuri11은 11px 기준 비트맵 폰트 — 글자 크기는 반드시 11의 배수(11/22/33/44)로.**
비정수 배율(12/14/16px)은 글리프가 뭉개진 뒤 임계처리되어 아웃라인과 합쳐져 가독성이 무너진다.
또한 작은 글씨(11px)는 아웃라인 1px 자동 적용(UISystem이 size<18이면 1px).
프리뷰 창처럼 캔버스가 ×0.75 등 비정수 **축소** 표시되면 11px 텍스트는 뭉개져 보이는데,
실기 모바일(×2+ 정수 배율)에서는 선명함 — 프리뷰만 보고 판단하지 말 것.

# 프로젝트 폴더 구조 — 어느 폴더에 무엇이 있는지 (2026-05-31 기준, 개요 파악용)

## 대원칙
- **엔진은 엔진, 게임은 게임.** 엔진 코드는 `engine/`, 게임은 `games/`.
- **게임 하나 = `games/<id>/` 한 폴더에 전부.** 로직·에셋·샘플 모두 그 안.
- **픽셀JSON·샘플 사진은 게임 간 절대 공유하지 않는다.** 공유 폴더 없음.
- **모래엔진 시그니처 기본 효과(모토)는 모든 신규 게임에 기본 탑재.** 명시 요청 전까지 제거 금지.

## 한 게임 폴더 구조 (`games/<id>/`)
```
games/<id>/
  game.json                 # 매니페스트 (해상도/레이어/씬/assetBase)
  index.html                # 단독 실행 진입점 (엔진은 ../../engine/ 에서 import)
  scenes/<scene>.scene.json # 씬 정의
  scripts/*.js              # 게임 스크립트 (모토 기본 효과 포함)
  palettes/*.json           # 이 게임 팔레트
  pixels/backgrounds/*.json # 배경 픽셀JSON (이 게임 전용)
  pixels/characters/*.json  # 캐릭터 픽셀JSON
  pixels/<category>/*.json  # 기타 카테고리
  samples/                  # 변환 전 원본 사진/GIF (이 게임 소스)
  project.json              # 에셋 매니페스트(카테고리/기본팔레트)
```
- `game.json` 의 `"assetBase": "./pixels"` → 에셋은 **자기 폴더 안**에서만 로드. = 자동 격리.
- `index.html` 은 엔진을 `../../engine/SandEngine.js` 로 import → 엔진과 분리, 게임은 독립.
- **단독 실행**: 개발서버에서 `/games/<id>/index.html` 열면 그대로 돈다 (에디터 불필요).
