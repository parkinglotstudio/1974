# 안티그래비티 필독 — 모래엔진 기본 상식과 흔한 실수

> 작성: Claude · 2026-06-08  
> **이 문서는 새 게임이나 기능을 만들기 전에 반드시 읽을 것.**  
> 엔진을 잘못 이해하면 코드가 동작해도 성능이 망가지거나 엔진 구조를 우회하게 된다.

---

## 1. 핵심 대원칙 (가장 중요)

> **엔진 = 로직(코드) / 게임 = 데이터(CSV·씬JSON·픽셀에셋)**

게임 코드는 엔진이 제공하는 인터페이스만 사용하고, 로직을 새로 짜지 않는다.  
엔진이 이미 하는 일을 게임 코드에서 다시 구현하면 → 버그, 성능 저하, 유지보수 불가.

---

## 2. 픽셀 데이터는 반드시 "변환 도구"로 만든다

### ❌ 절대 하지 말 것
```js
// 코드로 직접 픽셀 배열 생성 — 절대 금지
const pixels = [];
for (let y = 0; y < 252; y++)
    for (let x = 0; x < 252; x++)
        pixels.push([x, y, palIdx]);
entity._pixels = pixels;
```
이렇게 하면:
- JS에서 63,504개 배열 생성 = 느림
- 엔진 정적 최적화(한 번만 래스터) 완전 무효화
- 매 프레임 재처리

### ✅ 올바른 방법
픽셀 데이터는 **오프라인 변환 도구**로 JSON 파일로 만들어 두고, 씬에서 에셋으로 로드한다:
```
tools/gif_to_sprite.py       → 캐릭터 스프라이트 JSON
tools/png_to_pixels.py       → 배경/오브젝트 픽셀 JSON
tools/split_bg_layers.py     → 근경/원경 레이어 JSON
```
이렇게 만든 JSON을 씬에서 `asset` 필드로 참조하면 엔진이 알아서 로드·최적화.

---

## 3. 씬 로더를 게임 코드에서 재구현하지 말 것

### ❌ 잘못된 방법
```js
// PlayScene.js에서 씬을 수동으로 파싱하는 함수를 새로 만듦 — 금지
async applyCustomScene(engine, sceneJSON) {
    for (const def of sceneJSON.entities) {
        engine.entities.add(def.id, cfg);
    }
}
```
골목길의 `main.js`에 이미 `applyAssetScene()`이 완성돼 있다.  
씬 JSON 포맷을 지키면 엔진이 알아서 로드한다. 중복 구현 금지.

### ✅ 올바른 방법
씬 JSON(`scenes/main.scene.json`)만 올바르게 작성하면 된다.  
에셋 참조는 `asset` + `assetCategory` 필드로:
```json
{
  "id": "board_bg",
  "asset": "board_bg",
  "assetCategory": "objects",
  "layer": 1,
  "x": 9, "y": 84, "pw": 252, "ph": 252
}
```

## 4. 엔진 FX 초기화 블록을 절대 건드리지 말 것

`GolmokGame.js`의 `onEnter()` 상단에 있는 FX 초기화 블록:
```js
engine.lighting.setAmbient(0.0);
engine.glow.removeEmissiveIndex(10);   // ← 이거 지우면 밤에 캐릭터 노란 halo
engine.rim.disable();                  // ← enable()로 바꾸면 림이 2중 렌더됨
engine.fog.enableLayerFog({...});
engine.vignette.setPreset('warm');
```
이 블록은 **절대 삭제/수정 금지**. 이유가 주석에 설명돼 있다.  
2026-06-07: 이 블록을 삭제했다가 밤 halo·림 이중렌더·비네트 소실 발생. 직접 복구함.

## 5. 엔진 레이어 구조

| 레이어 | 이름 | 용도 | parallax |
|--------|------|------|----------|
| L0 | 원경(far) | 하늘·스카이라인 등 뒷 배경 | 0.30 |
| L1 | 근경(near) | 전경 실루엣·건물 등 앞 배경 | 1.00 |
| L2 | 게임 | 캐릭터·인터랙티브 오브젝트 | 1.00 (스크린 좌표) |
| L3 | UI | 점수·HUD 등 화면 고정 요소 | 1.40 |

- L2는 **스크린 좌표**로 렌더됨 (cameraX가 빠진 위치).
- L0·L1은 **월드 좌표** → cameraX에 parallax 곱해서 슬라이싱.

## 6. `_static` 최적화 — 정적 배경은 1회만 래스터

배경처럼 움직이지 않는 엔티티는 `_scanline` 포맷을 쓰면 엔진이 자동으로 정적 레이어로 처리 → 첫 프레임에만 그리고 이후 스킵 (= 성능 핵심).

동적 오브젝트(블록, 캐릭터 등)는 `pixels` 또는 `frames` 포맷 → 매 프레임 갱신.

## 7. doubledown_pixel이 잘못한 것 (구체적 사례)

`games/doubledown_pixel/` 폴더의 현재 코드에서 발견된 문제:

1. **`BoardRenderer.js`**: 블록마다 `entity._pixels`를 JS 코드로 동적 생성 → 변환 도구 사용 안 함, 성능 최악.
2. **`PlayScene.js`의 `applyCustomScene()`**: 씬 로더 재구현 → `main.js`의 `applyAssetScene()` 무시.
3. **`index.html`**: `console.log`마다 `/api/log`로 네트워크 요청 → 디버그 코드가 프로덕션에 남아있음. 반드시 제거.
4. **`hudExternalStore.js`**: React/Zustand 스타일 외부 상태 관리 → 엔진 데이터 흐름(CSV→씬→엔티티) 완전 무시.
5. **SceneJSON의 `board_bg` 엔티티**: asset 없이 씬에만 선언 → 실제 픽셀 데이터가 씬 JSON에 없어 빈 엔티티.

## 8. 새 게임 만들 때 올바른 순서

1. `games/새게임/game.json` 작성 (해상도·레이어·씬 목록)
2. 오프라인 도구로 픽셀 에셋 JSON 생성 → `pixels/` 폴더에 저장
3. `scenes/main.scene.json` 작성 (에셋 참조로 엔티티 배치)
4. `scripts/MyScene.js` 작성 — `Scene` 클래스 상속, `onEnter/onUpdate/onPostRender`만 구현
5. `index.html` 작성 — 골목길 `index.html` 구조 참고 (간단)
