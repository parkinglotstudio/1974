# ⚠️ DEV PROCESS IMPROVEMENT (오답 노트 & 개발 지침)

본 문서는 기획자의 HTML UI 가이드를 프로젝트에 이식하는 과정에서 발생했던 **과거의 실수들을 분석**하고, 향후 동일한 시행착오를 방지하기 위한 **강력한 개발 원칙**을 정리한 지침서입니다.

---

## ❌ 과거의 주요 실수 (Retrospective)

### 1. 시각적 디테일 경시 (Visual Negligence)
- **증상**: 원본 HTML의 세밀한 폰트 크기(px), 행간, 여백을 임의로 조정하여 가독성과 감성을 해침.
- **원인**: 기술적 구현(동작)에만 치중하고 기획자가 의도한 '디자인 픽셀 퍼펙트'의 중요성을 간과함.

### 2. 구조적 파편화 (Fragmented Porting)
- **증상**: 전체 HTML 구조를 가져오지 않고 부분적으로 코드를 이식하여 레이아웃이 깨지거나 스타일이 누락됨.
- **원인**: 원본 파일의 CSS 상속 관계와 DOM 구조를 충분히 분석하지 않고 성급하게 작업을 시작함.

### 3. 이벤트 핸들러 충돌 (Event Collision)
- **증상**: 새로 이식한 버튼이 작동하지 않거나, 기존 시스템의 클릭 이벤트가 소실됨.
- **원인**: `innerHTML`로 요소를 덮어쓰면서 기존에 바인딩 된 이벤트가 제거되는 현상을 고려하지 못함.

---

## ✅ 향후 UI 이식 5대 원칙 (The 5 Golden Rules)

### 1. [분석 우선] 100% 구조 파악
- 가이드로 제공된 HTML을 브라우저로 먼저 실행하여 레이아웃, 반응형 동작, CSS 우선순위를 완전히 분석한 후 코딩에 착수한다.

### 2. [디자인 준수] 1:1 픽셀 퍼펙트
- 폰트 종류, 컬러 코드, 소수점 단위의 여백까지 원본 가이드를 절대적으로 신뢰한다. 
- "더 나아 보일 것 같다"는 판단으로 기획 의도를 임의 변경하지 않는다.

### 3. [모듈화 이식] 네임스페이스 확보
- 이식되는 UI 스타일이 기존 프로젝트 스타일(`style.css`)과 충돌하지 않도록 전용 클래스나 ID를 사용하여 격리된 스타일링을 적용한다.

### 4. [안전한 바인딩] 이벤트 재정의
- `innerHTML` 교체 후에는 반드시 필요한 이벤트 핸들러를 다시 바인딩하거나, 위임(Delegation) 방식을 사용하여 인터랙션 누락을 방지한다.

### 5. [자가 검증] 원본 대조
- 작업 완료 후 기획자가 제공한 HTML 결과물과 실제 게임 내 화면을 나란히 놓고 시각적 차이가 없는지 최종 검수한 뒤 보고한다.

---

## 💡 기획자 협업을 위한 메모
- 기획자는 25년 경력의 베테랑이다. 그가 제공하는 HTML 프로토타입은 **이미 수많은 검증을 거친 '최종 시안'**임을 잊지 마라.
- 코드는 기술이지만, UI는 **사용자와의 약속(기획)**이다. 기술로 기획을 압도하려 하지 말고, 기획을 완벽하게 지원하라.
